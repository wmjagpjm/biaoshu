# P9C 真语义检索：产品决策门与基线审计

> **状态：** 等待用户产品决策；本文件不是代码实施授权。
> **范围：** P9C 的模型、数据出域、迁移、降级和验收门；不改动 P9A/P9B，不安装模型，不调用外部 API。

## 1. 目标

将知识库从“本地字符哈希向量 + 关键词”的历史基线，演进为可识别模型版本、向量维度、索引状态和降级状态的真语义检索。迁移必须可恢复，用户能区分“语义检索可用”“仅关键词可用”和“索引待重建”，不得把失败静默伪装成语义结果。

## 2. 已核实的当前基线

| 领域 | 当前事实 | 对 P9C 的影响 |
|---|---|---|
| 本地向量 | embedding_service 以 Python 内置 hash 构造 256 维字符/二元组向量。2026-07-14 在 PYTHONHASHSEED=0 与 1 的两个独立 Python 进程中，对同一文本得到不同摘要。 | 旧向量跨进程不可复现；不得把它标注为真语义模型或作为新模型的隐式回退。 |
| 外部向量 | 填写工作空间 embeddingModel 后，服务端会将完整分块文本和查询文本提交到 apiBaseUrl 的 /embeddings。任意异常或非成功响应均返回空结果并静默改用本地哈希。 | 是否允许正文/查询出域、外部服务归属和失败可见性必须先定。 |
| 元数据 | kb_chunks 仅保存 embedding_json，没有 provider、model、dimension、version、index status 或重建时间。 | 当前 cosine 在维度不一致时仅给 0 分；无法安全识别旧索引或支持回滚。 |
| 写入流程 | 重新分块时会先删除旧分块，再写入新向量。 | 现状不具备版本并存或失败后恢复；P9C 不能直接沿用这一迁移语义。 |
| 设置与密钥 | 现有工作空间设置会明文保存且 API 回显 apiKey；embeddingModel 只是自由文本输入。 | 如选择外部 API，必须明确接受此既有保密机约束，或将密钥托管/掩码改造纳入 P9C 范围。 |
| 本地运行时 | backend requirements 与当前 .venv 均未包含 sentence-transformers、torch、transformers 或 onnxruntime。 | 选择离线模型时，模型名称、许可证、文件来源、运行时、磁盘/内存/CPU 或 GPU 预算必须先冻结。 |
| 当前测试 | test_knowledge_rag 仅验证哈希向量的相对分数和基本混合检索，没有版本迁移、跨进程稳定性、真实语义评测或可见降级覆盖。 | 现有测试不能证明 P9C 已完成。 |

本审计不读取用户知识库内容、不运行重建、不请求任何外部模型；摘要差异仅来自固定测试文本和两个子进程。

## 3. 必须由用户确认的决策

| 决策 | 请选择或提供 | 不确认时的默认 |
|---|---|---|
| 向量来源 | 纯离线模型，或受控 OpenAI 兼容 API。 | 不实施；保留当前历史基线。 |
| 模型契约 | 供应方/模型标识、固定维度、中文招投标语料适配要求、许可证或 API 使用授权。 | 不下载模型、不发送数据。 |
| 数据出域 | 若为外部 API：可发送的字段范围、服务归属/区域、是否允许正文与查询出域、保留/训练条款。 | 视为禁止出域。 |
| 密钥责任 | 接受当前明文工作空间设置，或把密钥改为环境托管/掩码后再接外部 API。 | 不启用外部 API。 |
| 迁移策略 | 推荐“版本并存 + 可恢复后台重建 + 验收后切换”；或明确选择全量重建/长期保留旧索引。 | 不覆盖既有 embedding_json。 |
| 不可用行为 | 明确选择“显示不可用并停止语义检索”或“显示关键词降级”；禁止静默哈希回退。 | 显示不可用，不伪装为语义结果。 |
| 评测与阈值 | 脱敏评测集来源、人工相关性标注人、最少查询数、召回/排序阈值与允许的延迟/成本。 | 不调权、不宣称效果提升。 |

## 4. 规划约束与建议架构

以下内容仅在第 3 节全部确认后进入 Grok 的受限实施计划：

1. 将 embedding provider、model、dimension、version、index status 和生成时间保存为可查询元数据；旧 256 维哈希记录只能标为历史版本，不能混入新模型余弦比较。
2. 为每个工作空间维护可恢复的重建任务：新版本成功写完、验证通过后才切换活动版本；失败、取消或重启保留旧活动版本。
3. 查询侧只比较完全匹配的活动版本与维度；缺索引、版本不一致和服务不可用均返回可观察状态，按用户确认的政策决定是否附带关键词结果。
4. 外部 API 路径需要固定服务端配置、超时/限流/成本上限、脱敏日志和无密钥/正文回显；浏览器不得直接访问向量服务。
5. 离线路径需要锁定模型制品校验值、许可证、缓存目录、运行时版本和资源下限，并在未安装或模型不可载入时给出相同的可观察失败状态。
6. 权重调整只能依据脱敏评测集与记录的模型/索引版本；不得用单一演示问句或旧哈希测试代替验收。

## 5. 预期实施分段与验收标准

### 阶段 C1：版本化契约与可恢复重建

**目标：** 建立不破坏旧索引的版本元数据、重建任务、可见状态与失败语义。

**验收：**

- 相同文本在不同 Python 进程、不同重启后的同一已批准模型输出稳定且维度固定；
- 旧索引、新索引、维度不一致、重建中、重建失败和取消都不会给出伪语义分数；
- 重建失败后，旧活动版本仍可按已确认的行为提供检索；不丢失知识分块；
- 所有工作空间、任务和状态均隔离；日志/API 不含密钥、正文或供应方原始错误；
- 定向 pytest、后端全量、前端 lint/build，以及涉及后台任务时的本地 E2E 全部通过。

### 阶段 C2：脱敏评测与混合排序

**目标：** 用预先批准的评测集证明语义模型和混合排序的增益。

**验收：**

- 评测集不含客户机密、真实密钥或不允许出域的正文；
- 每条查询有人工相关性判断，可复跑并记录模型、索引版本、权重、耗时和成本；
- 达到用户确认的召回/排序阈值；未达标不得切换默认活动版本；
- 不可用行为、关键词降级（若批准）和 UI 提示有回归测试。

## 6. 本轮未完成项与下一步

本轮仅完成 P9C 开工审计和决策门文档，未产生代码、依赖、模型下载、索引重建或外部网络调用。收到第 3 节的明确决定后，Codex 将把已确认范围、文件白名单、非目标和验收命令写入独立实施计划，再交由 Grok 受限实现。
